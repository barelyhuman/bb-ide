bb connect + M4: Architecture Evolution

How bb went from a single-machine app to a remotely-reachable, multi-machine system — the tunnel, the auth model, and multi-host execution.

Internal engineering overview · staging on vibecodethis.site · generated 2026-07-04

The arc. bb started as a server + daemon bound to one machine, reachable only over localhost or Tailscale. This thread built bb connect: a thin Cloudflare cloud (accounts + a Durable-Object tunnel) that makes your own bb server reachable at <handle>.getbb.app from any browser, with your data never leaving your machine. Then it made that fast (edge caching, ~150ms→~25ms on warm asset requests), added a machine-credential pairing model so other machines can join as execution hosts, and finally removed the server-side single-host restriction so agent tasks can actually run on a chosen non-primary host.

Proven vs. remaining. Verified end-to-end on staging: browser access through the tunnel, edge caching, real GitHub login, two daemons connected simultaneously (trusted-network and through-the-tunnel), and a real thread.start executing on a second host. Remaining is product polish, not capability: a host-picker in the app UI and folding the tunnel machine-credential wiring into the daemon (a proxy harness stands in for it today).

1Where bb started

bb ships as a single-machine bundle. A launcher (or the Electron shell) spawns two processes on your machine: a server (owns the SQLite bb.db, the REST API, the client WebSocket, and serves the SPA) and a host daemon (runs provider CLIs, provisions workspaces, executes agent work). The daemon dials out to the server; the server serves the UI same-origin.

The security model was network perimeter: the server trusts whoever can reach it — localhost, or your Tailscale tailnet. There was no user auth on the API, and no way for a plain browser far from your machine to reach it. Two structural facts mattered for everything that followed:

  • The daemon already connects over a real, token-enrolled, reconnecting network protocol — it was never hard-wired to loopback.
  • host is a first-class entity and the data model already binds environments to a hostId; the codebase just funneled all execution to a single "primary" host.

2bb connect: the tunnel shipped to staging

The goal: reach your own bb from any browser, anywhere, without moving your data to the cloud and without Tailscale. The design keeps the cloud thin — it only does accounts + routing, never stores threads or code. Your server dials out and holds a tunnel; a Cloudflare Worker gate authenticates visitors and relays the whole origin down that tunnel.

The tunnel is built on Cloudflare Durable Objects: one DO per handle. The tunnel client (your machine) holds an outbound WebSocket to its DO; browser requests hit a Worker that authenticates and forwards them into the same DO, which frames them down the tunnel. Requests and WebSockets are multiplexed over the one socket with stream ids.

Why a Durable Object rather than a rented tunnel (ngrok / Cloudflare Tunnel / Piko)? Owning the protocol keeps the (thin) auth plane ours while renting the (hard) global infrastructure; DO WebSocket hibernation makes idle tunnels essentially free; and there is no per-tunnel account ceiling. The tradeoff — a single-threaded DO per handle — is fine for bb's text-and-terminal traffic and is largely sidestepped by the caching below.

3Tunnel speed: caching reverse proxy done + measured

The first working tunnel felt slow. Measured breakdown: a direct request to the local server was ~2ms; the same request through the tunnel was ~120–150ms. Two causes, both fixed.

Cause A — the gate did 3 sequential DB lookups per request

Every request (including each static asset) resolved the handle (2 queries) and verified the session (1 query) — three sequential edge→D1 round-trips before touching the tunnel. Fixes: collapse the handle+server resolution into one JOIN, and cache handle + session results per-isolate (short TTL) so a page's burst of requests mostly skips D1.

Cause B — everything traversed the tunnel, even cacheable assets

The gate became a proper caching reverse proxy (apps/connect/src/cache.ts). After owner-auth, cacheable GET responses are stored in Cloudflare's per-colo edge cache, keyed per handle, and served on repeat without touching the tunnel. Critically, it respects the origin's Cache-Control:

  • A production bb build serves immutable, content-hashed assets (max-age=31536000, immutable) → cached at the edge and in the browser.
  • bb's dynamic API responses carry no long cache directive → never cached (verified). No private data lands in the edge cache.
  • The dev server (Vite) works too: its module responses aren't long-cacheable → proxied fresh; its immutable dependency pre-bundles do get cached.
Measured result. On a warm keep-alive connection, repeat asset requests dropped from ~150ms → ~25ms (edge cache hit, no tunnel). Serving a bundled production build instead of Vite dev cut a page from hundreds of module requests to ~55; DOM-ready through the tunnel landed around ~200ms. WebSocket upgrades (bb's /ws, terminals) are never cached — they proxy directly.

before: ~150ms tunnel round-trip after (warm): ~25ms edge hit

4The better-auth cookie decoding fix bug fixed

Real GitHub logins were silently rejected with 401 at the gate, even though the login itself succeeded. Root cause: better-auth URL-encodes its session cookie value. The cookie is <token>.<base64-hmac-signature>, and the base64 signature's + / = arrive as %2B %2F %3D. The gate was comparing that URL-encoded value against a freshly-computed raw base64 signature — so it never matched. (My earlier curl tests passed only because curl sent the signature un-encoded.)

Fix: the gate now decodeURIComponent()s the cookie value before splitting and verifying the HMAC (apps/connect/src/session.ts). Verified with a real browser-shaped, URL-encoded cookie → 200.

Two smaller better-auth gotchas surfaced in the dashboard's sign-out and were fixed the same way: sign-out requires Content-Type: application/json (else 415) and a JSON body (an empty body 500s), plus the Origin header the browser supplies for its CSRF check.

5Machine credentials & the "Add a machine" flow built + verified at the gate

Connecting your server is one pairing. Connecting an additional machine (as an execution host) is a second, distinct pairing — and it must not become an open door on the internet. The design: a per-machine machine credential that authorizes a daemon's traffic to traverse the gate.

  1. Owner clicks "Add a machine" in the dashboard → bb-web mints a one-time machine-pair code (short TTL).
  2. The new machine redeems the code at /api/connect/redeem-machine → gets a durable machine credential (bbcm_…), stored hashed in D1 (plaintext lives only on the machine).
  3. The daemon presents the credential on an x-bb-connect-machine header for its traffic to the server through the tunnel.

New machine table (migration 0001_machine.sql) in packages/connect-db, distinct from the server row: the server is the bb server behind the tunnel; a machine is a daemon on another computer reaching it. The gate authenticates the credential; the credential only grants the right to traverse the tunnel to /internal — the bb server still performs its own host-key auth underneath.

6/internal forwarding & the defense-in-depth auth model verified

For a remote daemon to enroll and hold its session through the tunnel, the gate must forward bb's daemon protocol paths (/internal/hosts/enroll-key, /internal/hosts/enroll, /internal/session/*, /internal/ws). But these can't be forwarded blindly: the tunnel makes traffic appear to arrive at the server over loopback, which would let a random internet client mint enroll keys (the server's enroll-key issuance is loopback-gated). So the gate authenticates /internal/* itself, then strips the credential and forwards.

Gate behavior verified with curl against staging: no credential → 403; valid credential → the request reaches the server and gets the server's own 401 (proving it forwarded); bogus credential → 403; and an enroll key was successfully minted through the tunnel only when a valid machine credential was present — closing the loopback-trust hole.

7The direct / trusted-network daemon path verified

Not every multi-machine setup needs the tunnel. When your machines share a trusted network (LAN or Tailscale), a second daemon can point straight at the server's private address — no gate, no machine credential — because the network is the boundary. bb already supported this: the server binds all interfaces and the daemon takes an arbitrary BB_SERVER_URL. This path needed no new code; it just needed the single-host execution restriction lifted (§8).

Verified: a second daemon (test-host-2) with a fresh data dir enrolled and connected over loopback and appeared alongside the primary in GET /api/v1/hosts, both connected at once. A third daemon (tunnel-host-3) connected simultaneously through the tunnel — proving both connection models coexist.

8Multi-host execution routing — the last blocker implemented + verified

Daemons could connect, but tasks still couldn't run on them: the server funneled all execution through a single "primary" host and rejected any other with 400 unsupported_host ("Only the local host daemon is supported").

The investigation found this was the only blocker. The request schema, environment rows, and every dispatch layer (callHostOnlineRpc → hub.requestHostOnlineRpc, keyed by hostId) were already host-aware. A single validation gate, assertPrimaryHostId, was the choke point — used in two distinct roles:

Validate a supplied host
"this hostId came from the user / a stored environment — is it allowed?" → should accept any connected host.
Resolve a default host
"no host chosen — which one?" → should keep defaulting to the local primary.

The fix (smallest correct change): add assertUsableHostId — accept any non-destroyed public host (404 for unknown/destroyed, 400 for non-public) — and use it at the ~10 validate-a-supplied-hostId sites. The resolve-a-default helpers (resolvePrimaryHostId / requireConnectedPrimaryHostId) are untouched, so single-host behavior is unchanged. Liveness is enforced downstream at dispatch, exactly as before. A duplicate gate on the batch thread-create path (assertProjectDataPrimaryHostId) was removed in favor of the existing project-membership check. The CLI gained --host <id> on thread spawn; the API already carried hostId.

Proven live, not just connected. With a real second daemon enrolled, bb thread spawn --project proj_personal --host <host2> created a thread whose environment row bound to the second host, and the host-2 daemon executed it: Online host RPC … commandType: thread.start … ok:true. Regressions held: spawning without --host added an environment on the primary host; an unknown --host was rejected.

11What remains — honest status

The following are product polish, not capability gaps — the requested multi-machine execution works today via the API and CLI.
  • follow-up Daemon-side credential wiring. The bb-connect tunnel path was proven with a small header-injecting proxy harness (/tmp/machine-proxy.mjs) standing in for real daemon code that sends x-bb-connect-machine. Folding this into apps/host-daemon (server-client / enroll / WS) plus a bb connect machine launcher subcommand is the productization step. The trusted-network path needs none of this and is fully real today.
  • follow-up Host picker in the app UI. The wire (API) and CLI (--host) carry hostId, but apps/app currently passes only one host to its environment picker. Feeding it the full host list is the UI change; no server work needed.
  • by design The staging demo depends on local processes. The dev bb server, the bundled-build preview server, and the tunnel client run on this machine; if any stops, sawyerhood.vibecodethis.site shows offline. The wildcard-DNS + prod-domain cutover to real getbb.app is deferred (documented in plans/bb-connect-v1.md).
  • deferred Tunnel backpressure. Credit-based flow control in the wire protocol is stubbed; lower priority now that assets bypass the tunnel via the edge cache.

Honesty notes

  • Sessions used in some staging tests were forged against the real better-auth secret (HMAC-signed) to exercise gated paths without a browser GitHub login — the auth mechanism itself (OAuth URL generation, cookie signing/verification) is real and separately verified.
  • The tunnel latency floor (~120ms cold, first fetch) is inherent to the cloud round-trip; only cached/warm paths reach ~25ms. This is a relay, not peer-to-peer — it will never match Tailscale's ~1ms direct path (that would require a WebRTC/LAN direct path, explicitly out of scope).
