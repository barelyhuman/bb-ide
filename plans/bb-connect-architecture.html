bb connect

Make your own bb server reachable from anywhere via a new, separate bb cloud service (auth + tunneling at getbb.app) — with zero auth code in the core app and a one-line script to connect every machine you own. July 4, 2026 · v2 — auth moved out of core · companion to hosted-bb-architecture.html

TL;DR. The cloud piece is a new web service, not a change to bb: bb cloud owns accounts, login, the machine registry, and a relay fleet. Your bb server gains one sidecar-shaped component — a tunnel client — that dials out to the relay and injects authenticated remote traffic into the server over loopback. Because bb's server already trusts loopback, the core server, its routes, and the SPA are completely unchanged:

  • sawyer.getbb.app = your whole bb origin (SPA, /api/v1, /ws, terminals) relayed through the tunnel. Login happens at getbb.app; the relay only forwards traffic from sessions that own your tunnel.
  • Other machines: curl getbb.app/connect | sh installs a daemon that joins your server through the same URL — the existing host-daemon join flow; daemon auth (host keys) already lives in core and keeps working through the relay untouched.
  • Defense-in-depth without core changes: the tunnel client pins your account ID at connect time and rejects any relayed request not tagged with it — so a relay routing bug can't expose your server.

Net: one new deployable (bb cloud), one new client component (the tunnel client), no schema changes, no auth middleware, no frontend changes. bb stays local-first and fully functional with the cloud absent.

1Why auth doesn't belong in the core app

bb's security model today is network-perimeter trust: the server trusts whoever can reach it (loopback, or your tailnet). The obvious way to go remote is to replace that with in-app user auth — accounts, sessions, middleware on every route. But there's a cheaper move that keeps the model intact: don't widen the perimeter; extend it through exactly one authenticated party.

  1. The server never listens on the internet. The only remote ingress is the outbound tunnel that the tunnel client itself dialed. No open port, no new attack surface on the box.
  2. bb cloud authenticates everything before it enters the tunnel. Accounts, sessions, tokens — all owned by the new service. It forwards a request down your tunnel only after verifying the caller's account owns that tunnel, tagging each request with the account ID over the authenticated tunnel channel.
  3. The tunnel client verifies, then injects over loopback. It pinned your account ID at connect time; a mismatch is dropped. What it forwards to 127.0.0.1:38886 is, from the server's perspective, ordinary trusted local traffic. Core server: zero changes.
  4. Daemons were never the problem. /internal/* already self-authenticates with per-machine host keys (apps/server/src/services/machine-auth.ts) — the one place bb already has real auth, and it stays exactly where it is. The relay just passes it through.

Is that as safe as auth in the server? Honestly weighed:

ThreatAuth in core (better-auth users + sessions)Auth in bb cloud + tunnel-client pinning
Random internet scanning / direct hitsBlocked by auth middlewareBlocked harder — no listening port at all
Relay routing bug (wrong user's traffic down your tunnel)Blocked by server-side session checkBlocked by account pinning in the tunnel client
Stolen getbb.app session cookieFull access (server session was bootstrapped from the same login)Full access — identical exposure
Full relay compromise (attacker runs the relay)Not stopped — the relay holds the assertion signing keys either wayNot stopped — same. Only E2E encryption (roadmap, Q2) stops this, and it's available in both designs
Someone on your LAN hits :38886 directlyBlocked if auth enforced locally (but that breaks local-first UX)Same as bb today — loopback/tailnet perimeter, unchanged

The security delta is ~zero; the code delta is enormous. In-core auth means schema changes, session plumbing through every typed route, WS upgrade auth, an auth mode flag, a login UI in the product, and keeping local-first UX working around all of it. Cloud-owned auth means the core app doesn't know the cloud exists.

The one real cost: bb has no per-user identity inside the app, so future sharing/teammate features can't attribute actions or scope permissions per person. The escape hatch is cheap and deferrable: the tunnel client can forward a verified X-BB-Identity header, and the server starts reading it only when such a feature exists. Don't build it now (accepted-but-ignored fields are forbidden anyway) — just know the door is open.

2The model

  • bb cloud (new web service, getbb.app): accounts + login, handle → tunnel registry, connect-code issuance, and the relay data plane for *.getbb.app. A separate deployable with its own small DB. It has no bb product logic — it can't render a thread; it only routes.
  • Tunnel client (new, ships with bb): a component of the bb-app launcher (or sidecar process) on the server machine. Dials out, holds a multiplexed WS, verifies account pinning, proxies to loopback. The core server doesn't know it exists.
  • Your server: today's bb server + bb.db, byte-for-byte unchanged.
  • Your other machines: daemon-only installs joining via the tunnel URL.
The core trick (unchanged from v1): tunnel the entire origin. sawyer.getbb.app relays the SPA, /api/v1, /ws, terminal upgrades, and /internal/* alike. bb's same-origin frontend works unmodified, the UI version always matches your server, and daemons need nothing but BB_SERVER_URL=https://sawyer.getbb.app. New in v2: because the tunnel client injects over loopback, even the server's loopback-gated surfaces (like enroll-key issuance, apps/server/src/internal/hosts.ts:20) keep working for the owner with zero changes — the perimeter model is preserved end to end.

3How the key flows work

Flow A — Connecting your server (once)

  1. Sign up at getbb.app (bb cloud account: email or OAuth). Pick a handle → sawyer.getbb.app.
  2. On the machine running bb: bb connect. Opens the browser to getbb.app; you approve "Connect this server to sawyer.getbb.app" → bb cloud mints a one-time, short-TTL connect code.
  3. The tunnel client exchanges the code for a durable tunnel credential and pins the account ID; both stored under ~/.bb/ (same pattern as the daemon's auth.json). It dials the relay and holds a multiplexed WS — reconnect/backoff/heartbeat discipline copied from apps/host-daemon/src/server-connection.ts.
  4. getbb.app shows your server: online · last seen · version. Note the whole exchange touched only bb cloud and the tunnel client — the bb server process was never modified, restarted, or reconfigured.

Flow B — Opening bb from anywhere

  1. Visit getbb.app on any device, log in (bb cloud session), click your server — or go straight to sawyer.getbb.app; unauthenticated visitors get bb cloud's login page, never your server.
  2. The relay verifies the session's account owns this tunnel, then forwards the request down it, tagged with the account ID on the authenticated tunnel channel.
  3. The tunnel client checks the tag against its pinned account and proxies to 127.0.0.1:38886. The server sees a loopback request — trusted, as always.
  4. Everything else is transparent: SPA loads from your server, /ws streams thread events, terminals attach. Server offline → getbb.app renders "last seen 2h ago" instead of a broken page. CLI/SDK from remote machines authenticate to the relay with a bb-cloud personal access token (bbc_…), which the relay validates and strips before forwarding.

Flow C — Connecting your other machines

  1. In the bb UI (reached via the tunnel): "Add a machine" → one-liner: curl -fsSL https://getbb.app/connect | sh -s -- --code XXXX-XXXX. The join code comes from bb's own enroll-key flow — which the owner can mint because their traffic arrives over loopback via the tunnel client.
  2. The script installs the daemon and runs the existing flow: host-daemon join --join-code … --server-url https://sawyer.getbb.app (packages/bb-app/src/launcher.ts already resolves this). The daemon enrolls, gets its host key, and holds its reconnecting WS to /internal/ws — all through the relay, all with auth that already exists in core.
  3. The machine appears as a Host; environments target it; threads, terminals, file browsing reach it via server → tunnel → relay → daemon. Later: direct LAN fast path with relay fallback.
  4. Installed as launchd/systemd service; heartbeat-driven update nudges.

The bb cloud service itself

  • Auth plane: accounts, sessions, PATs, handles, connect codes, machine registry. This is where better-auth (or similar) lives now — in the new service, not in bb.
  • Data plane: terminates TLS for *.getbb.app (wildcard cert), maps subdomain → live tunnel, forwards HTTP/WS as multiplexed frames tagged with the authenticated account. Streams, never stores; no payload logging.
  • Multi-node: sticky-route by subdomain at the LB to start; Redis route table + cross-node forwarding only when scale demands.
  • Where it lives: new app in the monorepo (apps/cloud) so the tunnel protocol contract can be a shared package (packages/tunnel-contract) typed end-to-end with the tunnel client — same pattern as packages/host-daemon-contract. Deployed independently; the core server has no dependency on it.
  • Build vs buy: we very likely don't build the data plane for v1 — see the next section.

Build vs buy: build the protocol, host it on Workers + Durable Objects

Since bb-over-Tailscale is already validated in daily use, whole-origin remote access (WS, terminals, SPA) isn't in question — only the relay path is. Options considered, honestly:

OptionVerdict
Ship cloudflared (Cloudflare Tunnel per user) No — off-label. Cloudflare Tunnel is a Zero Trust product for connecting an organization's own infrastructure to its CF account. Embedding one tunnel per end-user of a consumer product under our account is not its intended shape, and the platform says so: ~1,000 tunnels per account. Plus we'd ship and lifecycle a ~50 MB third-party Go binary on every user's machine.
ngrok for platforms (@ngrok/ngrok embedded SDK) This is the commercial product intended for exactly this use case — tunnels embedded in your SaaS, JS-native, no sidecar binary. But pricing kills it for consumer fan-out: ~$15/mo per additional active endpoint (endpoint-hours; an always-on bb server is active nearly every hour) + $0.10/GB + $1/100k requests ⇒ up to ~$15/user-server/month in COGS, vs. pennies on Durable Objects. Also puts a vendor on the data path permanently.
Self-host an OSS tunnel server (Piko — multi-tenant, production-focused; frp, pgrok, zrok, Pangolin) Piko is genuinely designed for this (multi-tenant reverse tunnels, subdomain routing, node cluster, k8s-friendly). But it only replaces the relay host — the part Durable Objects makes nearly free — while leaving every cost we'd pay anyway: its upstream agent is a Go binary (same objection as cloudflared) with only a Go SDK, so a no-sidecar install means writing a TS client for their protocol ≈ the effort of writing ours; the auth gate is still ours to build in front; and we take on fleet ops (StatefulSet/VMs, LB, wildcard TLS, upgrades, DDoS on our own IPs). End-to-end WebSocket proxying is also unverified in their docs — a must-test for bb. Right role: the BYO-relay / Cloudflare-exit fallback, not v1.
Build our own on Workers + Durable Objects Yes — recommended. Own the (small) protocol; rent the (hard) infrastructure. Details below.

The Durable Object tunnel design

One Durable Object per tunnel, keyed by handle (sawyer). Two kinds of parties connect inbound to it, which is exactly the shape DOs are built for:

  1. Tunnel client (pure TypeScript, no sidecar): the launcher opens a WebSocket to its DO and holds it — using partysocket, the same reconnecting-WS library bb's frontend already uses. Auth: the durable tunnel credential from bb connect. No cloudflared, no Go binary, no third-party release cadence.
  2. Visitors: a request to sawyer.getbb.app hits the Worker (auth gate: bb cloud session or bbc_… PAT → account owns handle → attach account tag), which routes it to the DO. The DO frames the request down the tunnel WS; the tunnel client verifies the pinned account tag, proxies to 127.0.0.1:38886, and streams response frames back. Browser WebSockets (/ws, terminals) are carried the same way — WS-over-WS with stream IDs.
  3. The protocol is ours (packages/tunnel-contract): stream open/close, HTTP head, body chunks, WS frames, backpressure credits, account tag, heartbeat. Because we own the frames, the account-pinning check rides natively, and relay-blind E2E encryption (Q2) has a place to live later — neither was possible on rented data planes.

Why this is the right hosting substrate rather than a Node fleet:

  • Zero fleet ops: no VMs, no TLS management, no load balancer; wildcard *.getbb.app routing and global anycast come with the platform. The auth plane (accounts, registry, connect codes) is Workers + D1/KV alongside it — bb cloud v1 is nearly infrastructure-free.
  • Idle tunnels cost ~nothing: the WebSocket Hibernation API evicts an idle DO from memory while keeping both sockets open — duration charges stop while hibernating, and auto-response ping/pong handles keepalives without waking it. A thousand connected-but-quiet bb servers ≈ free.
  • No tenant ceiling: millions of DOs is normal usage; the per-user DO is a natural shard. Contrast the 1,000-tunnel cap.
  • Testable and portable: plain TS end to end, runs locally under workerd/miniflare in vitest; and since the protocol is the asset, the host is swappable — the same frames could later run on a Node relay fleet (or self-hosted for BYO-relay users) without touching the tunnel client or guard semantics.
Honest engineering scope: the real work is the multiplexing protocol — request framing, body chunking (WS messages cap at 32 MiB, so chunk ~1 MiB with credit-based backpressure), WS-over-WS piping, and reconnect-with-resume semantics. Sized in code, the data plane is small: tunnel-contract frames ~300–500 LoC, the tunnel DO ~600–1,000, the Worker gate ~200–400, the TS tunnel client ~500–800 — roughly 2,000–2,700 LoC of product code, with tests doubling it and the auth plane (accounts, registry, connect codes) adding ~1–2k more. Call it a few focused weeks, and backpressure is where the bugs will live. Known per-tenant ceilings: a DO is single-threaded, so one user's tunnel has bounded throughput — ample for bb's JSON + terminal text; large file downloads will be merely okay. Vendor coupling remains (Cloudflare hosts the DOs) but is thin and swappable, unlike owning neither protocol nor host.

The ⚠️ is the honest caveat: in v1 the relay terminates TLS and is in the trusted computing base — same trust level as ngrok or Cloudflare Tunnel, far less than a hosted control plane that stores everything. Roadmap: E2E-encrypted tunnel payloads (relay-blind), which this split uniquely enables since the SPA and the keys both come from your server, not the relay (Q2).

7Phased plan

Note what's gone relative to v1: the entire "add user auth to the server" phase. The critical path is now one new service and one new client component.

Phase 0 — Spike: minimal Durable Object tunnel ~a week

No core bb changes. Whole-origin remote access is already validated by daily bb-over-Tailscale use — this spike tests our protocol shape on the DO substrate.

  • Prototype the skinny version: one DO, one hardcoded handle, no accounts. A ~300-line tunnel-client script on a laptop connects out; a Worker routes spike.getbb.app to the DO; frames carry HTTP + one WS stream, naive chunking, no backpressure yet.
  • Drive a stock bb server through it from a phone on cellular: SPA load, live thread stream over /ws, terminal attach and feel, idle-then-resume after 10 minutes (hibernation + auto-response pings), and a host-daemon join from a second machine through the public URL. (Throwaway server — the spike has no real auth.)
Exit criteria: a written go/no-go note: observed latency, terminal feel, WS idle/hibernation behavior, upgrade handling, and where framing got awkward. If the DO path disappoints, fall back to self-hosting Piko/frp fronted by our auth proxy, same phase structure.

Phase 1 — bb cloud v1: the DO tunnel + auth plane medium-large · the new build, all ours, zero fleet ops

Touches: new apps/cloud (Worker gate + tunnel DO + auth plane), new packages/tunnel-contract (frame types shared by DO and tunnel client), bb connect + tunnel client in packages/bb-app launcher. Core server: untouched.

  • Protocol first: harden the spike's framing in packages/tunnel-contract — stream open/close, HTTP head, ~1 MiB body chunks with credit-based backpressure, WS-over-WS frames, account tag, heartbeat, reconnect-with-resume. Typed end-to-end, unit-tested against workerd/miniflare locally.
  • Auth plane: accounts (email/OAuth), sessions, PATs (bbc_…), handles, machine registry, one-time connect codes — Workers + D1/KV. All identity lives here, not in bb.
  • Data plane: Worker gate on *.getbb.app/* (session/PAT → account owns handle → tag → route to the handle's DO); tunnel DO with hibernation + auto-response pings; per-account rate limits at the Worker.
  • Tunnel client (pure TS, in the launcher): bb connect — browser approval → connect code → durable credential + pinned account → hold the DO WebSocket (partysocket discipline), verify account tags, proxy streams to 127.0.0.1:38886. Status in bb status and the UI footer. No sidecar binaries.
  • Lifecycle: "disconnect this server" from getbb.app revokes the credential and drops the tunnel; offline servers get a "last seen…" page served by the Worker; credential rotation from day one.
Exit criteria: from a phone on cellular, log into getbb.app, reach you.getbb.app, watch a live thread, attach a terminal — server on a laptop behind NAT, core bb at an unmodified commit from main. Auth matrix test in CI: no session, wrong account, revoked credential, expired code — rejected at the Worker; mistagged frames — rejected at the tunnel client. Protocol tests: chunked large response, backpressure under a slow reader, tunnel-drop mid-stream recovers cleanly after reconnect. Validate: real-network manual run + workerd-based integration suite in apps/cloud.

Phase 2 — bb connect for other machines + multi-host UX medium · mostly existing plumbing

Touches: install script (bb cloud serves it), launchd/systemd service wrapper, multi-host UX in apps/app — the one core-app workstream

  • "Add a machine" in the bb UI mints a join code via the existing enroll-key route (owner's traffic arrives over loopback through the tunnel — no route changes); curl getbb.app/connect | sh installs the daemon and runs the existing join flow against https://you.getbb.app.
  • Service install + heartbeat-driven update nudges.
  • Multi-host product UX: host picker on environments, online/offline surfaced, thread starts queued until the target host reconnects, per-host capability toggles (remote terminals / directory browsing), idempotent + retried daemon→server reporting so a dropped tunnel never loses events.
  • Optional, can slip: LAN-direct fast path with relay fallback.
Exit criteria: three-machine test — server at home, work laptop + cloud VM connected via the script — all visible as hosts; a thread runs on each from a phone; killing a daemon mid-thread yields recovered-or-cleanly-failed, never wedged. Validate: real multi-machine run + integration test with two daemons against one server through a local relay.

Phase 3 — Mobile medium · parallelizable after Phase 1

Touches: new packages/client-core, new mobile app

  • Extract apps/app/src/lib/{api,ws,query-client} into @bb/client-core with a platform shim; web adopts it lazily (nothing forces the refactor first).
  • Mobile v1 (Expo): bb cloud login → pick server → thread list, live thread view, permission-prompt approvals, push notifications for "agent needs you." Push fits this architecture cleanly: the tunnel client (which sees connection-level events) or the server posts a content-free ping to a bb cloud notification endpoint; thread content never rides the push payload.
Exit criteria: TestFlight build views a live thread and answers a permission prompt over cellular against a home server; push arrives when a thread blocks. Validate: manual TestFlight session; client-core adoption keeps web e2e green.

Later / out of scope for v1

  • Alternative hosting for the same protocol — a Node relay fleet or a self-hostable relay binary (BYO relay) — if Cloudflare costs/limits ever demand an exit; the frames, tunnel client, and Worker checks all carry over.
  • E2E-encrypted tunnel payloads (relay-blind) — Q2; straightforward to slot in later since the frames are ours.
  • Hosted bb servers ("we run your always-on machine") behind the same relay.
  • Identity plumb-through (X-BB-Identity) when a sharing/teams feature actually needs it.
  • Custom domains; WebRTC/LAN direct paths.

8Risks & open questions

Question / riskNotes & leaning
Q1 bb cloud is the single auth layer — a relay auth bug is game over for exposed servers. Accepted consciously (see §1 threat table: in-core auth wouldn't materially change this). Mitigations: account pinning at the tunnel client (covers routing bugs — the likeliest failure class), small and boring auth surface, adversarial auth-matrix tests in CI, connect-code TTLs, instant revocation, audit log of connects. Consider optional step-up (re-auth/PIN enforced at the relay) for terminal attach.
Q2 Relay terminates TLS ⇒ can observe traffic in flight (not at rest). v1 trust ≈ ngrok/Cloudflare Tunnel; publish the no-persistence policy. Roadmap: E2E payload encryption where browser and server share a key the relay never sees — decide how much this matters before building the custom relay, since it shapes the framing protocol.
Q3 The server must be on for bb to exist remotely. Inherent; be loud in UX ("last seen 2h ago"). Recommend an always-on box as "the server"; hosted-server tier later without architecture change.
Q4 No in-app identity: every relayed user (today: just you) is "the local user." Fine for a personal tool; blocks sharing features. The identity-header escape hatch is designed but deliberately unbuilt (no accepted-but-ignored fields). Revisit when teams become a real feature.
Q5 Tunnel client placement: inside the launcher process or a sidecar? Leaning launcher-integrated (one process tree, it already supervises server+daemon and knows the server port), with the proxy logic in a package so a standalone sidecar stays possible for non-launcher deployments.
Q6 DO-path engineering risks: backpressure correctness, per-tenant throughput (a DO is single-threaded), 32 MiB frame cap, hibernation behavior under real traffic. Backpressure is where the bugs will live — make it a first-class part of the protocol tests, not an afterthought. Per-tenant throughput is ample for bb's JSON + terminal text; large file downloads are merely okay (acceptable, documented). Chunking handles the frame cap. Hibernation + auto-response pings must be confirmed in the Phase 0 spike; idle cost rounds to zero when it works. Traffic is text-dominated and bursty, so egress economics are mild.
Q7 One server per account, or many? Start with one (one subdomain, one bb.db, zero picker UX) but make the registry schema N-capable from day one (home.sawyer.getbb.app) — cheap now, painful to retrofit.

Suggested first moves

  1. Phase 0 spike — the skinny DO tunnel (one DO, hardcoded handle, no auth) driving a stock bb server from a phone. ~A week, answers the remaining scary questions: terminal feel, hibernation/idle behavior, where the framing gets awkward.
  2. Sketch packages/tunnel-contract frame types (stream open/close, HTTP head, body chunk + credit, WS frame, account tag, heartbeat) — the spike should consume this file from day one so the protocol is real, not throwaway.
  3. Write the Q2 ADR (relay trust level / E2E ambition) — owning the frames makes E2E genuinely reachable, so decide whether v1 framing should reserve room for it (e.g. opaque encrypted payload mode).
