bb → Hosted Frontend + Installable Daemon

A plan for evolving bb toward the multica topology: the app lives at getbb.app, users install only a daemon on their machines, and mobile becomes just another client of the same backend. July 4, 2026 · sources: bb repo + multica repo (cloned & read)

TL;DR. bb is closer to this than it looks. The daemon already dials out to the server over a token-enrolled reconnecting WebSocket, host is a first-class entity in the data model, and a host-daemon join --join-code --server-url flow already exists in the launcher. Four things stand between today and getbb.app:

  1. The frontend hardcodes same-origin for API + WS (apps/app/src/lib/ws.ts:53).
  2. /api/v1 has no user auth — the security boundary is the network (localhost/Tailscale).
  3. Daemon enrollment trusts loopback (apps/server/src/internal/hosts.ts:20) instead of an account-authenticated pairing flow.
  4. The server + SQLite bb.db assume single-tenant, on the user's machine.

The plan closes these in order, each phase shippable on its own: (0) decouple frontend origin → (1) user accounts + API auth → (2) internet daemon pairing → (3) multi-tenant hosted control plane → (4) shared client core + mobile. bb stays local-first throughout: hosted is a deployment configuration of the same code, mirroring multica's cloud/self-host parity in reverse.

1How multica works

Multica is a hosted agent-orchestration product with the exact topology we want: a cloud backend + web app, a mobile app, and a single installable binary (multica) that is both the CLI and a daemon. Everything below is read from the source (Go backend at server/, clients under apps/ and packages/), not just the README.

The six mechanics that matter

1 · Outbound-only daemon, HTTP control plane, WS as a hint

The single most important design decision in the codebase. The daemon's correctness never depends on a live socket: it polls POST /api/daemon/runtimes/{id}/tasks/claim every ~3s and heartbeats every ~15s over plain authenticated HTTP (server/internal/daemon/client.go). A WebSocket to /api/daemon/ws (server/internal/daemon/wakeup.go) exists only to push daemon:task_available hints ("claim now instead of waiting for the next poll") and to carry heartbeats while connected. If the WS drops, polling silently carries the load. Streaming task output is likewise batched HTTP POSTs (POST /tasks/{id}/messages with sequence numbers), and terminal calls (complete/fail) are retried with backoff and idempotent server-side. NAT, flaky wifi, laptop sleep — none of it can corrupt state.

2 · Two separate WebSocket hubs

Daemon-facing (server/internal/daemonws/hub.go, header-token auth) and browser/mobile-facing (server/internal/realtime/hub.go, scoped fan-out to workspace|user|task|chat). A Redis relay lets any backend node push to any connected client, and it's optional — self-host runs single-node without it.

3 · Pairing = browser login mints a long-lived token

multica login starts a localhost HTTP listener, opens <app>/login?cli_callback=…&cli_state=… in the browser; after the user signs in, the web app redirects back to localhost with a short-lived JWT, which the CLI exchanges for a 90-day personal access token (mul_…) via POST /api/tokens (server/cmd/multica/cmd_auth.go). The daemon uses that PAT as a Bearer token on every request. A second token prefix (mcn_…) with an out-of-process verifier authenticates cloud-hosted runtimes — same protocol, different fleet.

4 · Runtimes and enqueue-time routing

Each daemon registers one runtime per (detected agent CLI × watched workspace). Routing is decided at enqueue time: an agent persona is bound to a runtime_id, and tasks inherit that binding (server/internal/service/task.go). There is no live scheduler — if the machine is offline, the task just sits queued until its daemon reconnects and claims it. Stale-dispatch reclaim and an orphan-recovery endpoint handle daemon crashes.

5 · No filesystem tunneling — access is task-scoped

The hosted UI never browses the user's disk. Code access is mediated by tasks: the daemon clones/worktrees registered repos locally, or runs in a local_directory pinned to a specific daemon. What flows back up is the task's message/tool-output stream, not filesystem RPC. This keeps the daemon strictly outbound-only.

6 · One codebase for cloud and self-host; one client core for all UIs

Cloud-only behavior is gated on env-var presence, not forks — self-host is docker-compose with Postgres + backend + web. On the client side, web, mobile, and desktop all consume packages/core (REST client, WS realtime layer, platform shim), which is why the iOS app was cheap to add.

2bb today

bb ships as a single-machine bundle: the bb-app launcher (or the Electron shell wrapping it) spawns a server (:38886 — SQLite bb.db, REST /api/v1, client WS /ws, daemon WS /internal/ws, serves the built SPA) and a host daemon (:38887 local API — runs provider CLIs, provisions workspaces). Other devices can browse to it over Tailscale, but they're control surfaces only.

5Where bb should follow multica — and where it shouldn't

Adopt from multica

  • Outbound-only daemon with resilience-first semantics. bb already dials out; adopt the rest of the discipline: idempotent, retried result delivery; last_seen_at staleness sweeper; orphan recovery on daemon restart; and queued-until-host-online semantics for thread starts targeting an offline host.
  • Browser-callback pairing. bb host join getbb.app → localhost listener → browser approval → short-lived join code → existing enroll flow. This replaces the loopback gate with an account-authenticated step and reuses everything downstream of it.
  • Two-hub separation. bb already has it (/ws client hub vs /internal/ws daemon hub) — preserve it through the auth work; don't merge.
  • Enqueue-time host binding. bb's environments already pin work to a host. Lean into that instead of building a scheduler: an offline host means the thread start waits, visibly.
  • One codebase, env-gated deployment. BB_AUTH_MODE=local|required, hosted features gated on config presence. No cloud fork.
  • Shared client core. Extract apps/app/src/lib/{api,ws,query-client} into @bb/client-core with a platform shim, exactly like packages/core in multica.

Deliberately keep bb's approach

  • Persistent duplex RPC to the daemon, not polling. Multica polls because its unit of work is a batch task. bb is interactive: terminals, permission prompts mid-run, live file browsing, editor-open. Those need a live server→daemon command channel, which bb already has (online-rpc.ts). Keep the WS as the primary channel; borrow multica's insight by making every state-changing daemon→server report (events, tool calls, thread completion) an idempotent, retryable delivery so a dropped socket never loses data — the socket is for liveness and interactivity, the durable writes are the source of truth.
  • Host as the unit of enrollment (vs. multica's runtime-per-CLI-per-workspace). One machine, one credential, providers discovered on it. Simpler and already built.
  • Richer host capabilities. Filesystem browsing and terminals through the daemon are core bb value. Multica avoided them; bb should keep them — they're safe because the channel is authenticated per-host and scoped per-user, but they raise the stakes on getting authz right (Phase 1) and deserve per-host permission toggles ("this host allows remote terminals: yes/no").
The one genuinely new subsystem is multi-tenancy, not networking. The wire protocols barely change. The work concentrates in identity (users, orgs, sessions), row-level scoping of every query, and the storage decision (§7-Q1).

6Phased plan

Each phase is independently shippable and useful. Phases 0–2 don't require hosting anything — they make bb hostable, and immediately improve today's Tailscale multi-device story. Phase 3 is where getbb.app goes live.

Phase 0 — Break the same-origin assumption small

Touches: apps/app/src/lib/api.ts, apps/app/src/lib/ws.ts, server CORS config

  • Introduce a single resolved serverBaseUrl in the app (runtime config: same-origin by default, overridable via a bootstrap config / VITE_BB_SERVER_URL), used by all fetches and the WS URL (ws.ts:53 currently derives from window.location.host).
  • CORS: extend the existing local-app-origins allowlist to include a configured hosted-app origin; cookies → SameSite=None; Secure when cross-origin (or prefer bearer tokens from day one, see Phase 1).
  • Add a minimal "connect to server" screen shown when no base URL is known — the seed of the getbb.app entry experience.
Exit criteria: the built SPA served from a different origin (e.g. npx serve dist on another port) fully drives a bb server — threads stream live over WS, terminals work. Validate: build app, serve statically on port X, point it at a dev server on port Y, run a thread end-to-end.

Phase 1 — User accounts & auth on /api/v1 medium

Touches: apps/server/src/server.ts middleware, better-auth config, packages/db schema, CLI/SDK token plumbing

  • Extend the already-installed better-auth from machine-keys-only to real users: email/OAuth sign-in, browser sessions, and user-scoped API tokens (bbu_… prefix) for CLI/SDK — mirroring multica's mul_… PATs. The user/apikey tables already exist in packages/db/src/schema.ts.
  • BB_AUTH_MODE=local (default): the server auto-provisions one local user and auto-authenticates loopback requests — zero UX change for today's users. BB_AUTH_MODE=required: every /api/v1 and /ws connection must present a session or token.
  • Scope authz at the project level first (owner-only), leaving org/membership for Phase 3. Every route handler gets the authenticated principal; add it to the typed route context in @bb/server-contract.
  • CLI: bb login <server-url> browser-callback flow (localhost listener → app login page → short-lived code → mint PAT), token stored per-server in client config.
Exit criteria: with BB_AUTH_MODE=required, every unauthenticated /api/v1 request and /ws upgrade is rejected (add an integration test sweeping the route table); with local, existing flows (desktop app, pnpm bb:dev, plugins) pass untouched. Validate: full integration-test suite + a manual bb login against a dev server from a second machine.

Phase 2 — Internet daemon pairing & multi-host medium

Touches: apps/server/src/internal/hosts.ts, packages/bb-app/src/launcher.ts join flow, host UX in the app

  • Replace the loopback gate on enroll-key issuance with an authenticated route: a signed-in user requests a join code (from the web UI: "Add a machine" → shows bb host join getbb.app --code XXXX; or CLI-initiated with browser approval). Short-lived, one-time, bound to the user — semantics the enroll token already has (internal/hosts.ts:36).
  • Hosts become user-owned rows; the daemon's host key inherits that ownership. Keep the loopback path only for BB_AUTH_MODE=local self-enrollment by the launcher.
  • Product surfaces for N hosts: host picker on environment creation, host online/offline in the UI, queued-until-online thread starts, per-host capability toggles (remote terminals, directory browsing).
  • Ship the daemon as a lightweight install: brew install bb / npx bb-app host-daemon join … — the launcher already resolves a join command; make it a first-class documented path with a login-item/launchd service.
  • Harden the daemon reporting path per §5: idempotent event/result posting with retry, orphaned-thread recovery on daemon restart, server-side last_seen_at sweeper.
Exit criteria: a daemon on machine B joins a server on machine A over the public internet (no tailnet), appears as an online host, and a thread runs on B driven from the web UI on A — including terminal attach and a mid-run permission prompt. Kill the daemon mid-thread; on restart the thread is recovered or cleanly failed, never wedged. Validate: two physical machines or a cloud VM + laptop; scripted in integration tests with two daemon processes against one server.

Phase 3 — Hosted control plane at getbb.app large

Touches: storage layer, tenancy model, ops/deploy; decision Q1 below gates this

  • Tenancy: orgs + memberships (better-auth has an organization plugin); every project/thread/host/environment row gains an org scope; row-level checks in the data layer, not per-handler.
  • Storage: resolve Q1 — port drizzle schema to Postgres for a shared multi-tenant DB, or run per-org SQLite (Turso/LiteFS-style). Postgres is the boring, multica-proven default; per-org SQLite preserves bb's "your data is one file" story and trivializes export/self-host migration.
  • Scale-out: WS fan-out relay (Redis pub/sub) so any server node reaches any client/daemon socket — optional exactly as in multica; single-node runs without it. Sticky routing for daemon sockets is a simpler interim.
  • Frontend: deploy apps/app to a CDN at getbb.app with the Phase-0 base-URL config baked to the hosted API; the "connect to server" screen also accepts self-hosted URLs, so one deployed frontend serves both hosted and BYO-server users.
  • Ops: rate limits, join-code abuse protection, audit log for host enrollment, secrets hygiene (provider API keys never leave the daemon machine — already true, keep it true), data export/delete.
Exit criteria: two unrelated accounts on one hosted deployment each pair a real machine, run threads, and can enumerate nothing of each other's (verified by an adversarial integration test hitting every route cross-tenant). A self-hosted server user can point getbb.app's frontend at their own server. Validate: staging deployment + cross-tenant probe suite + load test on the daemon hub.

Phase 4 — Shared client core & mobile medium, parallelizable after Phase 0

Touches: new packages/client-core, apps/app refactor, new apps/mobile

  • Extract apps/app/src/lib/{api,ws,query-client} + domain hooks into @bb/client-core with a platform shim (fetch/WS/storage injection) — multica's packages/core pattern. @bb/server-contract already gives it types.
  • Mobile app (Expo/React Native) consuming @bb/client-core: thread list, live thread view, permission-prompt approvals, push notifications for "agent needs you" — the killer mobile use case. No terminal/editor features in v1.
  • Desktop shell gradually becomes a thin wrapper over the same core, keeping only host-integration (spawn runtime, editor-open) native.
Exit criteria: the web app builds on @bb/client-core with no behavior change (existing e2e suite passes), and a TestFlight build can view a live-streaming thread and answer a permission prompt against a hosted or self-hosted server. Validate: e2e suite + manual TestFlight session against staging.

7Risks & open questions

Question / riskNotes & leaning
Q1 Storage for the hosted control plane: shared Postgres vs. per-org SQLite? Gates Phase 3 only. Postgres = proven (multica), simple ops, but a schema port + loses "your data is a file." Per-org SQLite = beautiful export/self-host story and keeps createConnection(":memory:") tests honest, but novel ops. Leaning: prototype the drizzle→Postgres port early to price it; decide on evidence.
Q2 Data residency: thread content (code snippets, diffs, terminal output) lands in the hosted DB. Real adoption blocker for some users. Mitigations: self-host parity as a first-class path (this plan keeps it free), org-level retention controls, e2e encryption as a future exploration. Be explicit in marketing.
Q3 Interactive latency through the cloud: terminal keystrokes and file browsing now hop laptop → cloud → laptop. Fine for the common case (phone → cloud → desktop at home). Worst case (browser and daemon on the same LAN, relayed via cloud) could later get a direct-connection upgrade (WebRTC/local discovery) — explicitly out of scope for v1.
Q4 Remote terminals + file browsing = high-value target once internet-facing. Per-host capability toggles (Phase 2), short join-code TTLs, host-key rotation (already supported by machine-auth), audit log, and rate limiting (Phase 3). Consider requiring re-auth for terminal attach on hosted.
Q5 Offline story: local-first users must never need the cloud. Non-negotiable: BB_AUTH_MODE=local keeps today's fully-offline single-machine bundle working forever. Hosted is additive.
Q6 Daemon distribution & updates. bb-app is already on npm and self-supervising; add brew + a bb host-daemon update path and a heartbeat-driven "update available" nudge (multica pushes CLI self-update actions through heartbeat acks — copy that).
Q7 Plugin system interaction: plugins currently trust a local single-user server. Plugins run server-side; on hosted they become tenant-scoped code execution — likely restrict to first-party plugins on hosted initially, keep full plugins for local/self-host.

Suggested first moves

  1. Phase 0 in one PR — it's small, immediately improves the Tailscale story, and de-risks everything after.
  2. Spike the drizzle→Postgres port (Q1) in a branch to get a real cost estimate before committing Phase 3's design.
  3. Write the Phase 1 auth-mode design as a short ADR — the local-mode auto-auth is the one place a wrong choice would degrade today's UX.
