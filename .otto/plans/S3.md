# Session S3 — Fork feature wiring (Tasks 8–11)

Spec: `.otto/specs/thread-forking-and-side-chats-e159.md` (Feature 1 — Fork;
Architecture table; OQ1 = Approach A; Data Model). Builds on S1 (server/domain:
`startedOnBehalfOf`, `childOrigin`, `canSpawnChild`, seed-without-run) and S2
(`MessageActionBar`, row identity forwarded into assistant message props).

## Resolved source facts (explored)

### hostId + branch name (Task 8)
- The source thread's environment is the canonical client source for both:
  - `hostId` = `Environment.hostId` (`packages/domain/src/environment.ts:82`).
  - branch name = `Environment.branchName`
    (`environment.ts:88`, nullable; the worktree's current branch).
  These are already available in `ThreadDetailView` via
  `useEnvironment(thread.environmentId)` (`environment` local, line ~557).
- provider/model/permissionMode: `providerId` is on `Thread`
  (`thread.ts:151`). `model`/`permissionMode` are NOT on the thread row — they
  live on `ResolvedThreadExecutionOptions`
  (`shared-types.ts:164`, fetched via `useThreadDefaultExecutionOptions`).
  So the fork builder takes resolved `{ model, permissionMode }` as inputs.
- **Fallback** when the source is NOT on a managed/worktree branch (no
  `environment`, no `branchName`, or a personal workspace): fall back to a
  managed worktree off `{ kind: "default" }` (server resolves the source's
  default branch) when we have a `hostId`; if there is no host at all
  (personal-only), the fork is not offered (the builder returns `null` and the
  Fork button stays disabled). This keeps the fork on a fresh worktree per
  spec while never producing an invalid request.

### Fork request body (Task 8) — pure builder `buildForkThreadRequest`
`apps/app/src/lib/fork-thread-request.ts`:
- input: `[{ type: "text", text: anchorMessageText, mentions: [] }]`.
- `startedOnBehalfOf: { initiator: "agent", senderThreadId: source.id }`.
- `parentThreadId: source.id` (satisfies the S1 boundary:
  senderThreadId === parentThreadId).
- `childOrigin: "fork"`.
- `environment`: `{ type: "host", hostId, workspace: { type: "managed-worktree",
  baseBranch } }` where `baseBranch` is `{ kind: "named", name: branchName }`
  when a branch is known, else `{ kind: "default" }`.
- `providerId` from source thread; `model`, `permissionMode` from resolved exec
  options; `title: "{source title} (fork)"`.
- Returns `AppCreateThreadRequest | null` (null when no host → not forkable).

### Fork create → navigate → focus (Task 9) — `useForkThreadFromMessage`
`apps/app/src/hooks/useForkThreadFromMessage.ts`:
- Models `RootComposeView.submitPrompt`: `useCreateThread().mutateAsync(body)` →
  `navigate(getThreadRoutePath({ projectId, threadId }))`.
- **Focus mechanism**: the thread-detail composer (`FollowUpPromptBox`) already
  passes `autoFocus`, and its `useLayoutEffect` refocuses keyed on
  `history.resetKey === thread.id` (`PromptBoxInternal.tsx:847`). So merely
  navigating to the freshly-created thread auto-focuses the composer once it
  mounts — no extra focus signal is needed, and because the composer is always
  mounted (even during `provisioning`), focus lands as soon as the editor
  mounts. This is the deferred-until-interactive behavior the spec asks for
  (Approach A seeds the thread idle; the user types the first turn). No
  `location.state.focusPrompt` plumbing into thread-detail required.

### Wire Fork button + depth guard + anchor icon (Task 10)
- The fork handler is created where thread+env+exec-options exist:
  `ThreadDetailView`. It is threaded into the timeline via a new
  `onForkMessage?: ForkMessageHandler` prop on `ThreadTimelinePane` →
  `ThreadTimelineRows` → `TimelineRendererStaticContext` (same pattern as
  `onOpenLink`). `ForkMessageHandler = (args: { messageText: string }) => void`.
- `ConversationMessageContent` consumes the forwarded row identity
  (`id/threadId/turnId/sourceSeqStart/sourceSeqEnd`, currently inert from S2):
  the assistant footer builds `onFork` from the static-context handler +
  `text`, passing it to `MessageActionBar`. `disabled` = active thread's
  `canSpawnChild === false` (read from static context, sourced from the
  ThreadResponse in ThreadDetailView). Leave `onSideChat` omitted for S4.
  Row identity becomes genuinely consumed (resolves the S2 review finding):
  the handler keys on the row, and `sourceTurnId` is forwarded to the builder.
- Anchor icon: `GeneratedConversationMessage` currently derives `leadingIcon`
  from `sourceKind` only. Add a `childOrigin` input so a fork anchor
  (`childOrigin === "fork"`) uses the `Fork` icon; otherwise the existing
  per-`sourceKind` icon. `childOrigin` of the *current thread* is threaded from
  the timeline static context (the rendered thread's response) into
  `UserConversationMessage` → `GeneratedConversationMessage`.

### Fork lineage UI (Task 11)
- `ThreadMetadataContent`: add a `ForksRow` listing child threads with
  `childOrigin === "fork"`, each a `Link` to the fork (reuse the parent-link
  pattern). Fetch via `useThreads({ projectId, parentThreadId: thread.id,
  childOrigin: "fork", archived: false })` — a targeted server query
  (`threadListQuerySchema.childOrigin` + `parentThreadId`), no load-all-filter.
  Requires extending the client `ThreadListFilters` + `listThreads` +
  `ThreadListQueryFilters` (query key) with `childOrigin`.
