## Task 12 — SPIKE result (resolves OQ4): YES, `PromptBoxInternal` can drive a non-route thread.

**Verdict: confirmed, no rework needed. Reuse `PromptBoxInternal` directly.**

Evidence (`apps/app/src/components/promptbox/PromptBoxInternal.tsx`):

- `PromptBoxInternal` is a **fully controlled** editor. It owns NO thread state:
  the caller supplies `value` / `mentionRanges` / `onChange` / `onSubmit`
  (`PromptBoxInternalProps`, lines 195–231). `onSubmit` is a bare `() => void` the
  caller wires to whatever turn-dispatch it wants — there is no implicit binding to
  the route's active thread.
- The thread-specific wrapper `FollowUpPromptBox` confirms this: it is the layer
  that knows about a thread (execution controls, queued messages, scroll anchor),
  and it itself just passes `value`/`onChange`/`onSubmit` down to
  `PromptBoxInternal`. None of that thread coupling lives in `PromptBoxInternal`.
- `autoFocus` + `history.resetKey` drive focus-on-mount (lines 847–853), so a
  side-chat composer mounted in a panel tab auto-focuses with no route involvement.

**Integration seam (Task 14):** `SideChatTabContent` owns local `value` /
`mentionRanges` state and renders `PromptBoxInternal` directly (NOT
`FollowUpPromptBox`, which is overweight: it assumes a route-active thread, queued
messages, context banners, scroll anchor). `onSubmit`:
- First submit (tab `threadId === null`): build the create-thread request (visible
  question + `agent-only` snapshot) and `useCreateThread().mutateAsync`, then set the
  tab's `threadId` via a new `setSideChatThreadId` updater.
- Later submits (`threadId !== null`): `useSendThreadMessage` against the child
  thread id (a normal follow-up turn, `mode: "enqueue"`-equivalent default).

Mentions config is supplied minimal/disabled in v1 (empty suggestions, no resolver)
— side chats are conversation-only; @-mention plumbing is a future enhancement and
not in scope. This keeps the composer reuse honest (no dead config).

## Task 14 — `openSideChat` + `SideChatTabContent` + snapshot util.

- `useThreadFileTabs.openSideChat({ sourceThreadId, sourceMessageText })`: upsert a
  `side-chat` tab (threadId null) titled from the source message (first line,
  truncated), activate it, open the panel. Plus `setSideChatThreadId({ tabId,
  threadId })`, `activateSideChatTab(tabId)`, `closeSideChatTab(tabId)`, and an
  `activeSideChatTab` selector + `sideChatTabs` list (all open side-chat tabs, for
  the keep-mounted deck) following the browser-tab precedent exactly.
- New `apps/app/src/lib/side-chat-context-snapshot.ts` (pure, tested):
  `buildSideChatContextSnapshot({ rows, sourceMessageText, windowSize = 3 })` →
  `PromptInput[]` of `visibility: "agent-only"` text parts. Flattens the timeline
  tree (recurse `turn.children`) into ordered conversation rows (`role`, `text`),
  takes the tail window (the source/anchor message + the `windowSize` preceding
  conversation messages + everything after the anchor in the current turn), and
  formats one `agent-only` text part describing the main-thread excerpt
  (`[bb side-chat context]\n{Role}: {text}` lines). One-time snapshot; does not
  refresh (documented spec limitation).
- New `apps/app/src/components/secondary-panel/SideChatTabContent.tsx`: renders the
  child thread's messages (once `threadId` set, via `useThreadTimeline` →
  `ThreadTimelineRows`/existing timeline view) above a `PromptBoxInternal` composer
  (per Task-12 seam). First submit builds the request via a new pure
  `buildSideChatCreateRequest` helper (input = `[visible question, ...snapshot]`,
  `parentThreadId = sourceThreadId`, `childOrigin: "side-chat"`,
  `startedOnBehalfOf: null`, `environment = { type: "personal" }`,
  `permissionMode: "readonly"`, provider/model from the source's resolved exec
  options) → `useCreateThread` → `setSideChatThreadId`.
- KEEP MOUNTED: a `SideChatTabDeck` rendered next to `browserDeck` in
  `ThreadSecondaryPanel`, mounting every open `sideChatTabs` entry, only the active
  one visible (`display:none` otherwise) — streaming survives tab switches exactly
  like `BrowserTabDeck`.
