## Task 7 — Extract `MessageActionBar` + forward row identity

### Code-state note (deviation from spec wording)
The spec/prompt describe an existing inline hover-revealed `CopyButton` under the
agent message-root `group` div in a `<div className="mt-1 flex justify-start">`.
In the current tree (`main` and this branch HEAD) the **agent message has NO copy
footer** — only the user message has a `CopyButton`, and it is always-visible
(no `opacity-0 group-hover:opacity-100`). The `group-hover:opacity-100` pattern
is not present anywhere in the timeline today. I implement the spec's target
state: agent footer gains copy + inert fork/side-chat slots (hover-revealed),
user footer keeps copy only.

### New file `MessageActionBar.tsx`
- Single-object props type `MessageActionBarProps`:
  `{ messageText: string; alignment: "start" | "end"; onFork?: () => void;
     onSideChat?: () => void; disabled?: boolean }`.
- Renders a footer row (`mt-1 flex items-center gap-2`, justify-start/end from
  `alignment`) containing:
  - `CopyButton` (reused primitive) — only when `messageText` non-empty.
  - Fork icon-only button (`Icon name="Fork"`) — only when `onFork` provided.
  - SideChat icon-only button (`Icon name="SideChat"`) — only when `onSideChat`
    provided.
- Fork/SideChat buttons styled like `CopyButton`'s button class
  (`inline-flex size-5 items-center justify-center text-muted-foreground
   hover:text-foreground`) plus the hover-reveal classes
  `opacity-0 transition-opacity group-hover:opacity-100 focus-visible:opacity-100`.
- `disabled` disables the fork/side-chat buttons (copy stays usable).

### Inert-button approach (CHOSEN)
**Omit `onFork`/`onSideChat` at the call sites for now.** Buttons render only when
their handler is provided, so the agent footer shows just copy until S3/S4 wire
the handlers. This is cleaner than rendering permanently-`disabled` placeholders
(no dead disabled buttons in the DOM, no confusing always-greyed affordance).
The `disabled` prop is wired through for S3/S4's depth-guard use.
