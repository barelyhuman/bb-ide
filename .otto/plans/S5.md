# Session S5 — Timeline scroll-position preservation

Spec: `.otto/specs/thread-forking-and-side-chats-e159.md` → Feature 3 +
Architecture "Scroll persistence" row.

## Problem (grounded)

The timeline scroll subtree is force-remounted on thread switch:
- `apps/app/src/views/thread-detail/ThreadTimelinePane.tsx:170` — `<PageShell key={threadId} scrollBehavior="bottom-anchor">`.
- `apps/app/src/views/thread-detail/ThreadDetailSecondaryContent.tsx:254` — outer `PanelGroup key={timeline.threadId}` (remounts the whole pane).

`BottomAnchoredScrollBody` (`apps/app/src/components/ui/bottom-anchored-scroll-body.tsx`)
inits `shouldStickToBottomRef=true` and calls `queueBottomRestore()` in its mount
effect, so a returning thread lands at the bottom. Prior position is lost.

Rows are wrapped by `<div data-timeline-row-id={row.id}>` in
`ThreadTimelineRows.tsx:1198`; these are descendants of the scroll area, so
`scrollArea.querySelectorAll("[data-timeline-row-id]")` finds them.

## Task 16 — Anchor store + capture + restore

### New store (in-memory, no persistence)
New file `apps/app/src/lib/thread-timeline-scroll-anchor.ts`:
- `export interface ScrollAnchor { rowId: string; offsetWithinRow: number; atBottom: boolean }`
  (shared type, not inlined).
- `threadTimelineScrollAnchorAtomFamily = atomFamily((threadId: string) => atom<ScrollAnchor | null>(null))`.
- Plain `atom`, NOT `atomWithStorage` (spec: in-memory only; a stale pixel offset
  across reloads is fragile).

### How threadId reaches the scroll body
`BottomAnchoredScrollBody` currently has no threadId. The only `bottom-anchor`
caller of `PageShell` is `ThreadTimelinePane`, which already has `threadId`.
Thread it through as an OPTIONAL prop on `PageShell` (`scrollAnchorThreadId?`) and
on `BottomAnchoredScrollBody` (`scrollAnchorThreadId?: string`). When absent
(every other `PageShell` caller — none use `bottom-anchor` today except the
timeline; the static path ignores it), capture/restore are disabled and behavior
is identical to today. This keeps the single bottom-anchor caller wired without
a dead flag: the timeline sets it, nobody else does, and the prop has real
semantic meaning ("persist scroll for this thread id").

`ThreadTimelinePane` passes `scrollAnchorThreadId={threadId}` to `<PageShell>`.

### Capture (continuous, throttled)
Inside `BottomAnchoredScrollBody`, when `scrollAnchorThreadId` is set:
- Read the current anchor-setter from a ref updated each render via
  `useSetAtom(threadTimelineScrollAnchorAtomFamily(scrollAnchorThreadId))`.
- Add a throttled handler invoked from the existing `scroll` listener (reuse the
  one `scroll` event already wired; add the capture call alongside
  `syncBottomStateFromScroll`). Throttle ~100ms via a timestamp ref + trailing
  timeout, so we always record the final resting position.
- Compute the top-most visible row: iterate `scrollArea.querySelectorAll(
  "[data-timeline-row-id]")`, pick the first whose `getBoundingClientRect().bottom`
  > `scrollAreaRect.top + 1`. `offsetWithinRow = scrollAreaRect.top - rowRect.top`
  (>= 0 — how far we've scrolled past that row's top).
- `atBottom` = `isScrolledNearBottom(scrollArea)`.
- Write `{ rowId, offsetWithinRow, atBottom }`. If no rows present, skip (don't
  clobber a good anchor with an empty one).

Capture must NOT run only on unmount — an unmount read races the `key=` remount.
Continuous capture guarantees the atom is current before teardown.

### Restore (on mount, by row id)
A mount `useLayoutEffect` (so it runs before paint, before the existing mount
effect's `queueBottomRestore`) keyed on `scrollAnchorThreadId`:
- Read the saved anchor once via `store.get` (use a ref holding the latest value
  captured at mount, read synchronously). If none, or `atBottom === true`, do
  nothing — keep default stick-to-bottom (covers live-streaming + leaving at
  bottom returns at bottom).
- If `atBottom === false`: set a `pendingRestoreRef` = the anchor and run a
  `applyRestore()`:
  - `el = scrollArea.querySelector("[data-timeline-row-id='" + CSS.escape(rowId) + "']")`.
  - If found: `scrollElementIntoView({ element: el, options: { block: "start" } })`
    (this already sets `shouldStickToBottomRef=false`, `setIsAtBottom(false)`,
    cancels queued restore — exactly the suppression we need), then nudge
    `scrollArea.scrollTop += offsetWithinRow` to restore within-row offset.
    Clear `pendingRestoreRef`.
  - If not found yet (async hydration): leave `pendingRestoreRef` set; the row
    will appear after content settles.

### Implementation refinements (vs. initial sketch)
- The mount layout pass can read stale (pre-hydration, zero-size) row geometry,
  so restore does NOT stop the first time the row is found. The pending restore
  stays alive and re-applies on each ResizeObserver settle until the computed
  scrollTop is STABLE across two consecutive applications (or attempts run out).
  This is the production-correct version of "re-apply after content settles".
- `findTimelineRowElement` iterates the `[data-timeline-row-id]` NodeList and
  matches `dataset.timelineRowId` rather than building a `CSS.escape`-based
  attribute selector — arbitrary row ids never need escaping, and `CSS` is not
  always present (jsdom).
- A deliberate user scroll during the settle window clears the pending restore
  (`syncBottomStateFromScroll`) so we never yank the user back to the anchor.

### Coordinating with stick-to-bottom (avoid the fight)
- The mount effect calls `queueBottomRestore()` only if `shouldStickToBottomRef`
  is still true. So restoring must flip `shouldStickToBottomRef=false` BEFORE that
  effect runs. `scrollElementIntoView` already does this. To guarantee ordering,
  do the restore decision in a `useLayoutEffect` placed in the file so it runs;
  but React runs the existing `useEffect` (mount listeners + queueBottomRestore)
  AFTER layout effects, so a `useLayoutEffect` restore that sets the ref wins.
  When the saved row isn't present at first layout, we must ALSO suppress the
  mount `queueBottomRestore`: set `shouldStickToBottomRef.current = false`
  immediately in the restore layout effect whenever a non-bottom anchor exists
  (even before the row is found), and set `isAtBottom=false`. Then re-apply the
  pending restore.
- Re-apply once after content settles: drive `applyRestore()` from the existing
  `ResizeObserver` callback while `pendingRestoreRef` is set (it already observes
  scrollArea + scrollContent and fires as rows hydrate). Cap re-applies (e.g.
  give up after a few settle frames / once found) and then, if still not found,
  fall back to bottom (re-enable stick-to-bottom + `scrollToBottom`). This is the
  spec's "if the row isn't present yet, re-apply once after content settles, else
  fall back to bottom."

### atomFamily keyed by threadId ⇒ A→B→A correctness
Because capture writes to `threadTimelineScrollAnchorAtomFamily(threadId)` and
restore reads the same key, fast A→B→A restores A's own anchor; B's capture lands
under B's key and never bleeds into A.

## Task 17 — Tests

(reuse its rAF + ResizeObserver mocks — true externals only; no mock of our
modules). Use `getDefaultStore()` from jotai to read the atom the component wrote.

`getBoundingClientRect` for scrollArea + rows (the jsdom no-layout pattern already
in the file).

Cases:
- (a) Capture: scroll mid-timeline (user intent → away), fire scroll, assert the
  atom for thread A holds the top-most visible rowId + atBottom=false. Then
  remount with the same threadId and assert restore calls scrollIntoView / sets
  scrollTop near the saved row and bottom state is "away".
- (b) Leaving at bottom: at-bottom scroll → atom atBottom=true; remount → stays
  "bottom", no row restore (stick-to-bottom preserved).
- (c) A→B→A: capture anchor for A (away), switch to B (different rows, capture B),
  switch back to A and assert A's saved rowId is restored, not B's (atom keyed by
  threadId). Assert by reading both atom keys and by the restored scrollTop.

Each test must fail if capture or restore is removed (assert real outcomes:
atom contents + resulting scrollTop / bottom-state).
