# Session S1 — Server & domain foundation (thread forking + side chats)

Spec: `.otto/specs/thread-forking-and-side-chats-e159.md`

## OQ1 investigation result (verdict: APPROACH A is feasible)

The thread-start "displayed turn" and the "provider run" are already two
separate steps in the create flow:

- `requestThreadProvision` (`thread-provisioning.ts:147`) writes the persisted
  `client/turn/requested` thread-start event — currently hardcoded
  `initiator: "user"`, `senderThreadId: null`. This is the row the timeline
  renders. It does NOT dispatch a provider run.
- `startThreadIfEnvironmentReady` (`thread-provisioning.ts:79`) is what calls
  `requestThreadStart` (`thread-lifecycle.ts:942`), which builds + dispatches
  the host `thread.start` command (the actual provider run).

So Approach A ("immediate create, display-only anchor") is supported: persist
the thread-start `client/turn/requested` event with `initiator/senderThreadId`,
then skip `requestThreadStart` and land the thread in `idle` after the
workspace-ready event. Transition `provisioning -> idle` is already allowed
(`ALLOWED_TRANSITIONS`). No deferred-create (Approach B) required.

The seam: thread `startedOnBehalfOf` carries the `initiator/senderThreadId`
into the event; a `seedWithoutRun` flag on the provision-context request payload
makes `startThreadIfEnvironmentReady` skip the provider dispatch and transition
to idle. Both derive from `startedOnBehalfOf` being non-null (a fork/side-chat
start always waits for the user), so there is no separate dead flag.
