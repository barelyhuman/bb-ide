# Missed QA Invariants

## Host RPC Response Type Correlation

- Bug/hotfix commit: `e0b3e0c2b` (`fix: validate host RPC response messages by command type`)
- Invariant missed: a host-RPC success response must validate its `result`
  against its own `commandType`, and a response that resolves a pending request
  must have the same `commandType` as that pending request.
- Why broad QA/manual runbook missed it: the happy-path manager and thread
  flows only exercise ordinary valid server/daemon messages generated by the
  current code paths. They do not generate malformed or cross-correlated
  protocol responses.
- Automated guard added: `apps/server/test/hosts/online-rpc.test.ts` now rejects
  a schema-valid `host.read_file` response that uses the request id for a
  pending `host.file_metadata` call. Existing contract tests reject
  commandType/result shape mismatches, and the websocket-boundary tests reject
  malformed response messages with `1008 invalid-message`.
- Manual guard: `qa/manual-runbook.md` includes a manager-hire bootstrap smoke
  that checks the user-facing path reaches `idle` and asks the operator to
  inspect server/daemon logs for protocol disconnects or host-RPC mismatch
  entries around that action. This is only a symptom check; malformed protocol
  invariants belong in automated boundary tests.
